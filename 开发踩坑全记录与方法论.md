# 飞书多维表格字段捷径 × Seedance 2.0 API 开发踩坑全记录与方法论

> 本文档记录了在开发「飞书多维表格字段捷径插件集成火山引擎 Seedance 2.0 视频生成 API」过程中遇到的所有坑、解决方案，以及总结出的方法论。任何 AI 或开发者都可以根据本文档从零复现并成功开发出类似功能。

---

## 一、项目概述

### 技术栈
- **运行环境**: 飞书多维表格字段捷径 FaaS（Node.js）
- **SDK**: `@lark-opdev/block-basekit-server-api`
- **目标 API**: 火山引擎 Seedance 2.0 视频生成 API（`ark.cn-beijing.volces.com`）
- **CLI 工具**: `block-basekit-cli`（飞书官方，非公开 npm 包，通过 `package.json` 依赖安装）

### 核心功能
用户在飞书多维表格中配置参数（API Key、模式、模型、提示词、参考图片/视频等），点击运行后自动调用 Seedance API 生成视频，将视频 URL 写回表格。

## 二、踩坑全记录（按发现顺序）

### 坑 1：飞书内部附件 URL 外部无法访问

**现象**: 将飞书附件的 `tmp_url` 直接传给火山引擎 API，API 报错无法下载。

**原因**: 飞书附件的下载链接（`internal-api-drive-stream.feishu.cn`）是内部签名的临时 URL，仅在飞书内网/鉴权环境下有效，火山引擎服务器无法访问。

**解决方案**: 在插件服务端先下载附件，再转码后传给 API：
- **图片**: 下载 → 转 base64 data URI → 以 `data:image/png;base64,...` 形式传给 API（✅ API 支持）
- **视频**: 下载 → 上传到免费临时文件托管 → 以公网 URL 形式传给 API（❌ API 不支持视频 base64）

**关键代码位置**: `src/utils/media.ts` → `downloadAndEncodeBase64()` / `downloadAndUploadVideo()`

---

### 坑 2：Seedance API 的 video_url 不支持 base64

**现象**: 将视频转为 base64 data URI 传给 API，报错 `reference_video must be provided as a web url`。

**原因**: Seedance API 的 `image_url` 支持 base64 data URI，但 `video_url` **只支持公网可访问的 HTTP/HTTPS URL**。这是 API 的设计限制。

**解决方案**: 将视频上传到免费的临时文件托管服务，获取公网 URL 后传给 API。

**选型过程**:
| 服务 | 结果 |
|------|------|
| `0x0.st` | ❌ 已下线 |
| `tmpfiles.org` | ❌ 返回 HTML 页面，非直接下载链接 |
| **`litterbox.catbox.moe`** | ✅ 返回直接下载 URL，支持 1h/12h/24h/72h 有效期，无需认证，支持 CORS |

**关键代码位置**: `src/utils/media.ts` → `downloadAndUploadVideo()`

---

### 坑 3：视频像素数超限（r2v 模式 ≤ 927,408 像素）

**现象**: 用户上传 1080p 视频（1920×1080 = 2,073,600 像素），API 报错 `video pixel count must be less than or equal to 927408`。

**原因**: Seedance 2.0 的 r2v（reference to video）模式对参考视频有像素数限制：width × height ≤ 927,408。

**解决方案**: 在上传到临时托管之前，用 ffmpeg 压缩视频分辨率：
```
等比缩放公式: scale = sqrt(927408 / (width × height))
目标尺寸: newWidth = floor(width × scale), newHeight = floor(height × scale)
确保宽高为偶数（ffmpeg 编码要求）
```

示例: 1920×1080 → 1284×722（927,048 像素 ✅）

**关键代码位置**: `src/utils/media.ts` → `compressVideoIfNeeded()` / `calculateScaledDimensions()`

---

### 坑 4：context.fetch 不是标准 Response 对象

**现象**: 调用 `response.arrayBuffer()` 报错 `response.arrayBuffer is not a function`。

**原因**: 飞书 FaaS 环境的 `context.fetch` 返回的不是标准 Web API `Response` 对象，而是自定义实现。不同版本的 SDK 可能提供不同的方法。

**解决方案**: 三级兼容回退：
```typescript
if (typeof response.buffer === 'function') {
  buffer = await response.buffer();           // node-fetch v2 风格
} else if (typeof response.arrayBuffer === 'function') {
  buffer = Buffer.from(await response.arrayBuffer());  // 标准 Fetch API
} else if (typeof response.text === 'function') {
  buffer = Buffer.from(await response.text(), 'binary'); // 最保守方案
}
```

**关键代码位置**: `src/utils/media.ts` → `downloadAndEncodeBase64()` / `downloadAndUploadVideo()`

---

### 坑 5：response.ok 在非标准 Response 上不可靠

**现象**: `if (!response.ok)` 判断在某些情况下误判，导致正常响应被当作错误。

**原因**: 飞书 `context.fetch` 返回的对象可能没有 `ok` 属性，或其值不符合标准语义。

**解决方案**: 综合判断 HTTP 状态码和 `ok` 属性：
```typescript
const statusCode = response?.status;
const isOk = response?.ok !== false && (!statusCode || statusCode < 400);
```

### 坑 6：API 参数位置错误（duration/ratio/watermark 不是 text 的一部分）

**现象**: 将 `duration`、`ratio`、`watermark` 等参数拼接在 prompt text 中（如 `--dur 5`），API 行为不符合预期。

**原因**: 根据官方文档，这些参数应该作为**请求体的顶层字段**，而不是拼在文本提示词中。

**正确格式**:
```json
{
  "model": "doubao-seedance-2-0-260128",
  "content": [{ "type": "text", "text": "一只猫在跳舞" }],
  "duration": 5,
  "ratio": "16:9",
  "watermark": false,
  "generate_audio": true
}
```

**关键代码位置**: `src/api/createTask.ts` → `buildRequestBody()`

---

### 坑 7：飞书 SingleSelect 返回值格式不一致

**现象**: 表单中 SingleSelect 组件的值有时是字符串 `"text2video"`，有时是对象 `{ value: "text2video", label: "文生视频" }`。

**原因**: 飞书 FaaS 在不同场景（调试模式 vs 生产模式）下，SingleSelect 的返回值格式可能不同。

**解决方案**: 统一标准化函数：
```typescript
const normalizeSelect = (val: any): string => {
  if (!val) return '';
  if (typeof val === 'string') return val;
  if (val && typeof val === 'object' && val.value) return String(val.value);
  return String(val);
};
```

**关键代码位置**: `src/index.ts` → `execute()` 中的 `normalizeSelect()`

---

### 坑 8：域名白名单遗漏

**现象**: 下载飞书附件时请求被拦截，无响应。

**原因**: 飞书字段捷径有域名白名单机制，需要在代码中显式声明允许访问的域名。除了火山引擎 API 域名外，还需要添加飞书内部附件下载域名。

**解决方案**:
```typescript
basekit.addDomainList([
  'ark.cn-beijing.volces.com',           // 火山引擎 API
  'internal-api-drive-stream.feishu.cn'  // 飞书附件下载
]);
```

**注意**: 如果有多个入口文件（如 `index-status.ts`、`index-attachment.ts`），每个文件都需要添加域名白名单。

### 坑 9：CLI 命令混淆（dev vs start）

**现象**: 运行 `npx @lark-opdev/block-cli dev` 报 404；运行 `npm run dev` 报 `Can not find test/index.ts file`。

**原因**:
- `@lark-opdev/block-cli` 不在公开 npm 上，无法通过 npx 直接调用
- `npm run dev` = `block-basekit-cli dev:field`，这个命令用于运行单元测试（需要 `test/index.ts`）
- 正确的开发命令是 `npm start` = `block-basekit-cli start:field`（编译 src/ + 启动 HTTP 服务器端口 8080）

**解决方案**:
```bash
npm start     # ✅ 正确：编译 + 启动开发服务器（端口 8080）
npm run build # ✅ 编译生产版本
npm run pack  # ✅ 打包输出 zip
```

### 坑 10：FieldType.Attachment 作为 resultType 的问题

**现象**: 将 `resultType` 设为 `FieldType.Attachment`，生成的视频链接无法正确回写到表格。

**原因**: 飞书附件字段有复杂的上传协议（需要先获取上传凭证、分片上传等），直接写入 URL 字符串不生效。

**解决方案**: 使用 `FieldType.Text` 作为 `resultType`，将视频 URL 以纯文本形式存入表格。用户可复制链接到浏览器观看。

```typescript
resultType: {
  type: FieldType.Text,  // ✅ 用文本类型存储视频 URL
}
```

---

### 坑 11：FieldSelect 返回的是附件数据而非 URL 字符串

**现象**: 使用 `FieldComponent.FieldSelect` 选择附件列后，获取到的值不是 URL 字符串数组，而是附件对象数组。

**原因**: `FieldSelect` 配合 `supportType: [FieldType.Attachment]` 时，飞书直接传入该列的附件数据对象（包含 `tmp_url`、`type`、`name` 等字段），而不是 URL。

**这是好事**: 不需要额外调用飞书 API 获取附件信息，数据已经直接可用了。

**关键代码位置**: `src/utils/media.ts` → `extractImageAttachments()` / `extractVideoAttachments()`

---

### 坑 12：模板字面量不能用于 computed property key

**现象**: 代码 `[`===附件 ${i+1} 下载失败`]` 编译报语法错误。

**原因**: JavaScript 不允许在对象字面量的 computed property key 中直接使用模板字面量。

**解决方案**: 先提取到变量：
```typescript
const logKey = `===附件 ${i + 1} 下载失败`;
debugLog?.({ [logKey]: { fileName, error: String(e) } });
```

---

### 坑 13：fs.rmdir 的 deprecation 警告

**现象**: 启动时出现 `DeprecationWarning: fs.rmdir(path, { recursive: true }) will be removed`。

**原因**: 这是 `block-basekit-cli` 内部使用了已弃用的 Node.js API，不影响功能，但日志中会显示警告。

**解决方案**: 这是 CLI 工具本身的问题，插件代码无法修复。可忽略。

## 三、成功方法论总结

### 方法论 1：先读官方文档，再写代码

**教训**: 最初将 `duration`/`ratio`/`watermark` 拼在 text 中，导致行为异常。后来仔细阅读了全部 5 份官方文档才发现这些应该是顶层参数。

**原则**:
1. 开发前必须阅读 API 的**请求示例**和**参数说明**
2. 特别注意参数的**位置**（顶层 vs 嵌套在某个字段中）
3. 注意参数的**类型**（布尔值 vs 字符串 `"true"`/`"false"`）
4. 不同模型的能力和限制可能不同（如时长范围、支持的模式）

### 方法论 2：内网 URL → 外网可访问的转换模式

**核心问题**: 飞书/企业微信等平台的内部附件 URL 无法被外部 API 访问。

**通用解决方案**:
```
内网附件 URL → 服务端下载 → 编码/上传 → 外网可访问形式
```

| 媒体类型 | 编码方式 | 适用场景 |
|---------|---------|---------|
| 图片 | base64 data URI | API 支持 base64 时 |
| 图片 | 上传到图床获取公网 URL | API 不支持 base64 时 |
| 视频 | 上传到文件托管获取公网 URL | API 通常不支持视频 base64 |
| 音频 | 上传到文件托管获取公网 URL | 同视频 |

### 方法论 3：ffmpeg 视频压缩标准流程

当 API 对视频有分辨率/像素数限制时：
```
1. 下载视频 → 写入临时文件
2. ffprobe 获取宽高
3. 如果 width × height > 限制:
   a. scale = sqrt(限制 / 当前像素数)
   b. newWidth = floor(width × scale)，取偶数
   c. newHeight = floor(height × scale)，取偶数
   d. ffmpeg -i input -vf scale=W:H -preset ultrafast -crf 30 output
4. 上传压缩后的文件
5. 清理临时文件
```

### 方法论 4：飞书 FaaS 环境兼容性原则

飞书 FaaS 的 `context.fetch` 不是标准 Fetch API，需要遵循以下兼容原则：

1. **Response 方法**: 使用 `.buffer()` → `.arrayBuffer()` → `.text()` 三级回退
2. **ok 检查**: 不要只依赖 `response.ok`，要同时检查 `status` 码
3. **SingleSelect 值**: 始终用 `normalizeSelect()` 标准化
4. **域名白名单**: 所有外部请求的域名都必须在白名单中声明
5. **错误处理**: 用 try-catch 包裹所有外部调用，单个附件失败不应阻塞整体流程

### 方法论 5：容错设计模式

```
任务创建: 最多重试 2 次，间隔 3 秒
  - 不可重试错误（InvalidParameter 等）→ 立即返回错误信息
  - 可重试错误（网络波动、限流等）→ 自动重试

轮询查询: 渐进式间隔 + 连续失败熔断
  - 前 30 次: 10 秒间隔
  - 30~60 次: 15 秒间隔
  - 60 次后: 20 秒间隔
  - 连续失败 5 次: 提前终止，避免无效轮询
```

### 方法论 6：调试日志策略

飞书字段捷径的调试只能通过 `console.log`（会被 CLI 捕获并展示），因此：
1. 每个关键步骤都要有日志（下载、编码、上传、请求、响应）
2. base64 数据要截断显示（避免日志爆炸）
3. 错误信息要包含上下文（文件名、URL、状态码）
4. 使用结构化对象而非纯字符串（CLI 会 JSON 格式化展示）

## 五、关键 API 接口参考

### 创建任务
```
POST https://ark.cn-beijing.volces.com/api/v3/contents/generations/tasks
Authorization: Bearer {API_KEY}
Content-Type: application/json

{
  "model": "doubao-seedance-2-0-260128",

    { "type": "text", "text": "提示词" },
```
